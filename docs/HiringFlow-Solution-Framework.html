③ Test Case Generation Engine — Deep Dive (Core Skill)
🧪

testcase-generation / SKILL.md — The Core Intelligence Engine

The single most critical component of HiringFlow QA. It bridges the gap between a business user story and a fully executable, traceable, and standards-compliant automation test suite. It operates in two sequential phases, each with strict quality gates.

📖 Phase 1 — Requirement Understanding

  • Reads the user story provided by the user (file, pasted text, or attached document)
  • Extracts only functional acceptance criteria relevant to the story
  • Identifies all actors: Admin1, Recruiter1, Recruiter2, Candidate1–3
  • Maps each AC to an API contract using real POJO field names (never guessed)
  • Determines HTTP method, endpoint, request payload, expected response, and pre-conditions
  • Chooses generation mode: Mode 1 (single E2E) or Mode 2 (per-AC + E2E)
  • Ignores generic infrastructure concerns (auth, missing fields) — focuses on business logic

🧩 Phase 2 — Test Case Artifact Generation

  • Reads all POJO classes, URLGenerator, and ActorHelper to extract real API contracts
  • Reads userDetails.properties to bind correct user prefixes to actors
  • Creates a structured test case JSON file with complete request/response contracts
  • Immediately runs TestCaseGenerator to produce the structured test artifact
  • Verifies artifact row count matches JSON step count before proceeding
  • Enforces generic naming rules — never real person names in any content
  • Gate enforced: code generation blocked until both artifacts exist and are verified

Test Case Generation Modes

ModeTriggered WhenOutputE2E?
Mode 1 "Create high-level test cases" / "One test case" Exactly 1 E2E test case covering all ACs in a single flow ✅
Mode 2 "Cover all scenarios" / "All possible test cases" One test case per AC + business negatives + 1 final E2E ✅

⑤ Return on Investment
01
Cross-Functional Test Coverage
The Test Case Generation Engine interprets high-level acceptance criteria and automatically derives cross-functional scenarios — spanning multiple actors, roles, and API flows in a single test case. Coverage that would take days to design manually is produced in minutes.
02
Increased Test Coverage
Every acceptance criteria is mandatorily mapped to at least one test case. Happy path, business negative, role-based, and E2E scenarios are all enforced by the skill rules — leaving no AC untested and significantly raising overall coverage metrics.
03
Reduces Manual Errors & Misses
Payload fields are always sourced from real POJO classes — never guessed. Generic role-based naming is enforced. Gate checks block downstream steps if artifacts are missing or incomplete. Structural rules eliminate the most common test design errors at the source.
04
Accelerates E2E Automation
From user story to a fully compiled, auto-executed Java test class in one autonomous pipeline run. The agent handles: requirement analysis → test case creation → code generation → test execution → reporting → PR — all without manual intervention between steps.
05
Eliminates Manual Test Writing Effort
Test cases, request/response contracts, step descriptions, and test data are all generated automatically from the user story. QA engineers shift focus from writing boilerplate to reviewing and validating — dramatically reducing the time-to-automation per user story.
06
Project-Specific Test Documentation
Every generated test case artifact is traceable back to a specific acceptance criteria and stored as a versioned, structured artifact. This doubles as living documentation of the project's API contracts, business rules, and test strategies — always up to date.
